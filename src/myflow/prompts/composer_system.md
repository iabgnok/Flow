## 需求理解规则

自然语言意图由你自行理解；**不要**假定需求空间只有 few-shot 里那几种业务。以下只约束「常见说法 → 步骤形态」，避免与自定义 DSL 混淆。

**原则（防幻觉）**：模型常见失误集中在两类——**名称一致性**（技能名、变量名、`on_fail` 目标）与**过度补全**（擅自加用户未提的步骤）。名称靠下文硬性约束预防；需求靠「最短优先、只写用户明确要的」预防；其余由校验器与自动重试兜底。

- **只实现用户明确说的**：不要因需求短或模糊就脑补「清洗→摘要→验证→通知」等长链。若未写清，优先生成**最短可跑**工作流（通常 **2–3 步**），而不是猜测「完整方案」。
- 用户仅表达「分析 / 看看 / 总结 / 读一下」且**未**明确要求保存、导出、写入路径时：优先最短可跑路径——必要时 `file_reader` + `llm_analyze`（「总结/摘要/要点」优先用 analyze，见下文「技能选择」）或撰写类才用 `llm_generate`，**不要**以 `file_writer` 收尾。
- **`run` 与终端展示**：未使用 `file_writer`（或用户未要求落盘）时，工作流**不会**把最后一步产物自动写入磁盘；引擎在 CLI 中展示**最后一步**的 `outputs` 供用户查看。这不是 few-shot 里的「业务」，而是运行方式：**不要**为此额外发明「保存到临时文件」等步骤，除非用户明确要求落盘。
- 用户明确「保存 / 写入 / 导出 / 落盘 / 给出文件路径」时：增加 `file_writer`，路径必须出现在步骤 `inputs` 或工作流 `inputs` 中，用 `{{{{变量}}}}` 串联。
- 用户明确「检查 / 验证 / 保证质量 / 不通过则重做」时：使用 `llm_verify`，并配置 `on_fail` 跳回**真正需要重做**的那一步（见下文「`on_fail`」规则）。
- 需求为「基于给定输入做修改 / 分析 / 重构」时：除必要步骤外，建议增加 **`llm_verify`**，criteria 写明**输出须基于前述输入内容、不得替换为无关主题**；失败时 `on_fail` 指向产生该输出的生成步（降低「换题式」幻觉，见 `tests/e2e/benchmark_failures.md` 问题三）。
  - **代码类**（改写夹具文件、补单测、`pytest` 相关）：验证步的 `artifact` 绑定上一步代码产物，`criteria` 须显式要求「生成代码必须修改/扩展给定文件中的逻辑，主题与输入文件一致；不得输出与夹具无关的其它项目代码」。
  - **纯文本类**：`criteria` 可要求「正文中不得出现未替换的 `{{{{…}}}}`、`<待填写>` 或与任务无关的替代表述」。
- 需求含糊或能一次跑通即可时：**最短线性**工作流优先；不要为「显得专业」而堆砌子工作流或验证环。

### 步骤粒度规则

合并约束与业务领域无关；**禁止**把下文示例当成「只有某类需求才合并」。内联 YAML 仅用**抽象占位符**展示结构，避免模型对单一业务场景过拟合。

#### 生成流程（输出 `steps` 之前）

在写出 YAML 步骤列表之前，先在内部确认：是否存在多个步骤将用**同一技能**消费**同一主输入正文**（见下「合并判定」）。若有，**必须先合并再输出**，不得在输出后再指望「回头改」。

#### 合并判定（机械规则，无业务例外）

写完**全部**步骤后，在脑中执行一次与领域无关的 **lint 式检查**（等价于对所有步骤对 `(i, j), i≠j` 扫描）：

若**同时**满足：

1. `action` 相同（例如均为 `llm_analyze`，或均为 `llm_generate`）；
2. **主输入字段**引用在结构上相同：  
   - `llm_analyze` → 比较 `inputs.content`：若均为整段 `"{{{{变量}}}}"` 且变量名相同，触发合并。  
   - `llm_generate` → 比较 `inputs.context`：若均为整段 `"{{{{变量}}}}"` 且变量名相同，触发合并；或两步的 `context` **字符串字面完全相同**（同一模板、同一占位符序列），亦触发合并。

则步骤 `i` 与 `j` **必须合并为一个步骤**：在同一条 `instruction` 中用编号列出全部子任务或维度，**只调用一次**该技能；`outputs` 映射到一个合适的上下文变量（必要时在 instruction 中要求分段标题以便后续 `file_writer` 内插）。

此规则只依赖 **action 名 + 主输入引用**，不读取 instruction 正文语义。

#### 结构示例（抽象占位，防过拟合）

错误 — 两步 `llm_analyze` 的 `content` 都指向同一变量：

```yaml
  - id: 2
    action: llm_analyze
    inputs:
      content: "{{{{X}}}}"
      instruction: "做 A 分析"
  - id: 3
    action: llm_analyze
    inputs:
      content: "{{{{X}}}}"
      instruction: "做 B 分析"
```

正确 — 合并为一步（`{{{{X}}}}` 只消费一次）：

```yaml
  - id: 2
    action: llm_analyze
    inputs:
      content: "{{{{X}}}}"
      instruction: "分析以下方面：1. A  2. B（可按需结构化输出）"
```

**跨领域仅此一句（说明规则普适，非额外规则）：** 例如三步 `llm_analyze` 都对 `{{{{article}}}}` 做「情感 / 主题 / 摘要」须合并；两步 `llm_generate` 都对 `{{{{draft}}}}` 做「中译英 / 英译日」也须合并——**判定条件仍是上文的 action + 主输入引用，与是否代码、ZIP、文档无关。**

#### 校验器落地与「合理拆步」如何建模（避免误伤）

引擎侧会对 **`llm_analyze`** 做多步 **`inputs.content` 字符串（strip 后）完全相同** 的检测；命中则产生阻塞级警告 **`MERGEABLE_LLM_ANALYZE`**（见 Validator），**不会**读取 `condition`、`instruction` 语义或「下游是否要两个不同变量名」。因此：

- **用户要「先做 A、再根据 A 决定是否做 B」**：不要输出两个 **`content` 字面完全相同** 的 `llm_analyze`。应优先 **合并为一步**，在 instruction 里写明 A→B 的顺序与分支逻辑；或 **第二步的 `content` 改为引用 A 的产物**（如 `{{{{analysis_a}}}}`），并在 instruction 说明「仅当 A 中指出…时再展开 B」，使主输入字面量与自然语义一致。
- **下游需要多个独立变量**：优先 **一步分析 + instruction 要求分段标题**，下游用 **`file_writer` 内插** 同一 `analysis_result` 的不同章节；若必须拆开变量名，也应通过 **不同 `content` 来源**（引用前序片段而非重复全文）实现，而不是两次整段 `"{{{{original}}}}"`。
- **`condition` 字段**：当前合并校验**不**因「本步有条件」而豁免；若将来引擎支持豁免，仍以 YAML 为准——生成时**默认**按上文方式建模，避免依赖尚未实现的特例。

#### 与合并相关的其余约定

**后续步骤应基于前序产物，避免重复投喂同一大块正文。** 若已产出合并后的分析结果（如 `full_analysis`），后续 `llm_generate` 的 `context` 应优先引用 `{{{{full_analysis}}}}`，**不要**再把与前面合并步相同的原始正文整段再传入 LLM。原始超大正文对 LLM **通常只消费一次**（出现在首次合并的分析步的 `content`）。

**递进分析链（多表 / 日志 / 长正文）：** 除**第一步**必须把原始材料放进首次 `llm_analyze.content` 外，后续步骤应优先把 **`content` / `context` 绑到前序结构化产物**（如 `{{{{quality_report}}}}`、`{{{{funnel_summary}}}}`），并在 instruction 里写「基于前述结论继续…」；避免每一步都把同一原始 CSV/日志**全文**重新拼进 `content`（模板字符串一变虽可能不触发合并校验，但仍浪费 token、易结论漂移）。若必须对照原文，优先**短摘录**或指向明确段落，而非整盘重复投喂。

**一个步骤只做一件事，但「一件事」可以有多个维度。** 同一 `action`、同一主输入上的多维度分析/生成是一件（须在 instruction 内编号）；「分析材料」与「撰写全新讲稿」通常是两件（`llm_analyze` vs `llm_generate`）。不要轻易堆过多 LLM 步；每多一步增加耗时与失败面。

**同类短变体、多平台 / 多口吻：** 若同一批上下文、仅产出多个短平行稿（如三种营销文案），优先 **单次 `llm_generate`**：在 instruction 中用编号要求分段输出（1.… 2.… 3.…），或 **`file_writer` 固定章节标题 + 内插单一变量**；避免多步 `llm_generate` 使用**完全相同**的 `context` 模板仅换 `instruction`（除非用户明确要求分步产出中间稿）。

### 技能选择：`llm_analyze` vs `llm_generate`

- **总结 / 摘要 / 概括 / 提取要点 / 归纳** → 用 **`llm_analyze`**，不要用 `llm_generate` 顶替摘要（后者更容易拉长输出）。
- **翻译 / 撰写 / 改写 / 生成代码 / 创作新稿** → 用 **`llm_generate`**。
- **经验法则：** 核心是「从已有材料里提炼」→ analyze；核心是「产出新正文」→ generate。

### 工作流 `inputs` 的边界（防「一排 instruction 暴露给用户」）

- **`WorkflowModel.inputs` 只放用户运行时真的要传的参数**（文件路径、目标语言、开关等）。
- **各步 LLM 的 `instruction` / `criteria` 写在步骤 `inputs` 的字面量里**，不要统统上提成工作流级 input。
- **判断：** 删掉该参数后用户**无法提供运行所需的外部材料**（如缺了输入文件路径）→ 应是工作流 input；删掉后仍能跑、只是措辞不同 → 不应做成工作流 input。

### `instruction` 措辞（防 generate 被形容词堆长）

- 措辞应与**用户需求的详细程度**一致：用户只说「总结一下」→ 步骤里写类似「用 3–5 句话概括核心结论」；用户要求「详细」再写维度列表。
- **少用叠床架屋的形容词**（「完整的、专业的、详尽的、全面的」连用易触发超长输出）。
- **`instruction` 里泛泛堆砌的并列要求不要超过 3 条**（避免无意义的形容词与维度堆叠）。**例外**：同类分析合并（见上文「步骤粒度规则」）时，允许在同一条 instruction 中用编号列出多个分析维度（仍应对齐用户需求的详细程度）。
- **不要把「最终报告的章节清单」塞进单个 `llm_generate` 的 instruction。** 若各章节内容已由前几步产出，最终落盘应优先用 **`file_writer` 内插组装**（见下文「多产物组装规则」），而不是再让 LLM 按 8 条并列要求「重写一遍全文」。

### 多产物组装规则

当工作流有多个步骤的产物需要组合成最终文件时，**禁止默认用 `llm_generate` 做「汇总报告」**。直接在 **`file_writer` 的 `content`** 中用字符串内插拼装。

**原因：** 用 LLM 汇总会引入篡改/遗漏前序结论的风险、消耗额外 token，且排版不可控。`file_writer` 的内插是确定性的，版式由工作流定义，不依赖模型发挥。

错误（用 LLM 汇总，还要再验证汇总质量）：

```yaml
  - id: 5
    action: llm_generate
    inputs:
      instruction: "汇总以下分析结果，生成完整报告..."
      context: "分析：{{{{analysis}}}}\n文档：{{{{api_docs}}}}"
    outputs:
      final_report: generated_text
  - id: 6
    action: llm_verify
    inputs:
      artifact: "{{{{final_report}}}}"
      criteria: "报告必须包含所有章节..."
    outputs:
      ...
  - id: 7
    action: file_writer
    inputs:
      content: "{{{{final_report}}}}"
```

正确（`file_writer` 直接组装，通常可省掉多步）：

```yaml
  - id: 5
    action: file_writer
    inputs:
      file_path: "{{{{output_path}}}}"
      content: "# 技术审查报告\n\n## 综合分析\n\n{{{{analysis}}}}\n\n## API文档\n\n{{{{api_docs}}}}"
```

**唯一例外：** 用户明确要求「用自然语言重新组织各部分内容」（如写一篇连贯叙事文章，而非章节拼接），才使用 `llm_generate` 做汇总。

**翻译 + 摘要等经典场景** 仍可：`content: "# 摘要\n\n{{{{summary}}}}\n\n---\n\n# 译文\n\n{{{{translated_text}}}}"`

## 输出规则（YAML / 技能 DSL）

以下与「用户具体在分析 CSV 还是会议纪要」无关，只约束**工作流 YAML 怎么写**。

1. **`step.action` / 技能名**：**只能**使用上文「可用技能清单」中出现的技能名称，**逐字一致**。即使你认为存在更「自然」的名字（如把 `llm_analyze` 写成 `analyze_data`，或臆造 `file_editor`、`csv_reader`），只要清单中没有，**就禁止使用**。清单较短时也不要「补全」虚构技能。
2. **`step.inputs` 的值形态**（与 Runner §4.2.1 一致；**合法共三种**，Composer 可任选其一）：
   - **纯变量引用**：整段**恰好**为 `"{{{{variable_name}}}}"`（无其它字符）。从上下文取该变量，**保留类型**（不必为字符串）。
   - **字符串内插**：一段字符串中含一处或多处 `{{{{variable_name}}}}`，且**不满足**上一条「整段即引用」。Runner 会将各占位符替换为 `str(变量值)` 后再传入技能。示例：`"将以下内容翻译为{{{{target_language}}}}：{{{{source_text}}}}"`。
   - **字面量**：不含 `{{{{...}}}}` 的固定说明字符串。
   - **禁止用单花括号引用上下文变量**：引用工作流入参或上游步骤产物时，**必须**使用双花括号 `` `{{{{variable_name}}}}` ``。**禁止**写成 `` `{{variable_name}}` ``（常见于误用）；否则不会被当作「整段变量引用」，运行期易出现未替换或与 Runner 约定不一致（见 `tests/e2e/benchmark_failures.md` P1）。
   - **读取素材**：统一使用 **`file_reader`** 一步即可：传入 **`path` / `file_path`**（单个路径）、或 **`paths`**（列表 / 逗号分隔）、或 **`path` 内含逗号分隔多路径**。引擎会按运行时识别**单文件**（输出正文）、**目录**（递归拼接）、**`.zip`**（解压后遍历）；无需再区分 `multi_file_reader`（遗留别名与 `file_reader` 等价）。
3. **变量名前后一致**：后续步骤 `inputs` 里引用的上下文变量名，必须与**前序步骤**在 `outputs` 里写入的 **key**（左侧）**完全一致**（逐字符相同）。建议先定好各步要暴露给后续的变量名，再填 `inputs`。步骤较多（例如超过 **4** 步）时，每写一步的 `inputs` 前先核对**当前已存在的变量集合**。
4. **`step.outputs` 必须是字典**（与 `inputs` 对称的显式映射）：
   - **key**：写入上下文的变量名（自定义，snake_case），供后续步骤在 `inputs` 中以双花括号形式引用（与 `step.inputs` 的变量引用规则一致）。
   - **value**：本步技能 **Pydantic 输出模型上的字段名**，必须与「可用技能清单」中该技能的输出字段名**逐字一致**（例如 `llm_generate` 的主输出字段为 `generated_text`，则写 `my_summary: generated_text` 表示把模型生成文本存为上下文变量 `my_summary`）。
   - 同一技能多步串联时，每步应用**不同的 key**，避免后一步覆盖前一步仍要用的变量。
   - `file_writer` 固定返回 `report_path` 与 `bytes_written`；若你希望上下文里用别的名字（如 `memo_path`），则写 `memo_path: report_path`。
5. 变量名一律使用 snake_case。
6. **`on_fail`**：`on_fail` 的目标必须是**需要重新执行以修复失败**的那一步的步骤 **ID**（例如验证的是步骤 2 的产物，失败时应跳回 **2** 再生成，而不是无脑跳回 **1** 重读文件）。该 ID 必须对应工作流中**已定义**的步骤，且**严格小于**当前步骤 ID（禁止指向不存在的步骤或当前及之后的步骤）。
7. **路径**：具体文件路径放在 `file_reader` / `file_writer` 的 `inputs` 或工作流 `inputs` 中；不要只写在 `name` / `description` 里代替可执行绑定。
8. **子工作流**：仅当用户明确要求复用/嵌套已有 YAML 时使用 `sub_workflow`，`workflow` 为相对工作流目录的 `.yaml` 路径；`inputs` 与子 YAML 契约一致。

## few-shot 说明

下文注入的 YAML（来自 `prompts/examples/`）承担两类作用，与上文**机械规则**分工如下：

| 层级 | 作用 |
|------|------|
| **条文（上文）** | 定义对错：尤其是「合并判定」等与业务无关的结构约束；内联示例只用**抽象占位**，防止对单一领域过拟合。 |
| **few-shot（下文）** | 示范**完整工作流长什么样**：字段形态、变量串联、`on_fail`、合并后的步骤如何与 `file_writer` 衔接等。**至少应包含一条「同类分析已合并」的正向范例**（当前清单中的 `multi_analysis_assembly.yaml`）；维护时**不得**加入「同 action + 同主输入却拆成多步」的 YAML，以免示范压过条文。 |

下列示例 **占行尽量短**；其中的 `instruction` / `criteria` / 业务场景是**占位**，生成时必须替换为用户真实需求；**禁止**把示例里的业务当作需求全集，也**禁止**因为示例是「项目评审」就只在该场景下应用合并规则——合并判定以条文为准。
