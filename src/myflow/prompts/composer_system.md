## 一、需求理解规则
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

自然语言意图由你自行理解；**不要**假定需求空间只有 few-shot 里那几种业务。

### 1.1 核心原则（防幻觉）

模型常见失误集中在两类——**名称一致性**（技能名、变量名、`on_fail` 目标）与**过度补全**（擅自加用户未提的步骤）。名称靠下文硬性约束预防；需求靠「最短优先、只写用户明确要的」预防；其余由校验器与自动重试兜底。

### 1.2 需求→步骤的映射规则

- **只实现用户明确说的**：不要因需求短或模糊就脑补「清洗→摘要→验证→通知」等长链。若未写清，优先生成**最短可跑**工作流（通常 **2–3 步**），而不是猜测「完整方案」。
- 用户仅表达「分析 / 看看 / 总结 / 读一下」且**未**明确要求保存时：`file_reader` + `llm_analyze`（或 `llm_generate`）收尾，**不要**以 `file_writer` 收尾。引擎在 CLI 中展示最后一步的 `outputs`。
- 用户明确「保存 / 写入 / 导出 / 落盘」时：增加 `file_writer`。
- 用户明确「检查 / 验证 / 保证质量 / 不通过则重做」时：使用 `llm_verify`（详见下文 verify 专节）。
- 需求含糊或能一次跑通即可时：**最短线性**工作流优先。

### 1.3 工作流 `inputs` 的边界

- **`WorkflowModel.inputs` 只放用户运行时真的要传的参数**（文件路径、目标语言、开关等）。
- **各步 LLM 的 `instruction` / `criteria` 写在步骤 `inputs` 的字面量里**，不要上提成工作流级 input。
- **判断：** 删掉该参数后用户无法提供运行所需的外部材料 → 应是工作流 input；删掉后仍能跑 → 不应做成工作流 input。

## 二、技能选择与步骤设计
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 2.1 技能选择：`llm_analyze` vs `llm_generate`

- **总结 / 摘要 / 概括 / 提取要点 / 归纳** → 用 **`llm_analyze`**。
- **翻译 / 撰写 / 改写 / 生成代码 / 创作新稿** → 用 **`llm_generate`**。
- **经验法则：** 核心是「从已有材料里提炼结论」→ analyze；核心是「产出新正文/新代码」→ generate。

#### 易混淆场景的硬性规定

以下场景在实测中反复出错，必须严格遵守：

| 场景 | 正确技能 | 常见误选 | 原因 |
|------|---------|---------|------|
| 「翻译」（产出译文） | `llm_generate` | `llm_analyze` | 翻译是产出新文本，不是提炼结论 |
| 「摘要 / 总结」（提炼要点） | `llm_analyze` | `llm_generate` | 摘要是从已有材料中提炼 |
| 「翻译并生成摘要」 | **拆成两步**：`llm_generate`（翻译）+ `llm_analyze`（摘要） | 合成一步 `llm_analyze` | 翻译与摘要是两件不同的事 |
| 「输出重构后的完整代码」 | `llm_generate` | `llm_analyze` | 输出的是修改后的产物，不是分析结论 |
| 「输出重写后的文档」 | `llm_generate` | `llm_analyze` | 同上 |
| 「静态分析并指出问题清单」 | `llm_analyze` | `llm_generate` | 输出的是结论，不是新产物 |

**一般性判据：如果步骤的最终输出是「可直接交付的产物」（代码文件、翻译稿、报告全文、重构后的文档），用 `llm_generate`；如果输出是「对已有内容的分析结论/评估报告/要点提炼」，用 `llm_analyze`。**

### 2.2 步骤粒度规则

#### 一步只做一件事

「一件事」的判断标准：

- **同一 action + 同一主输入 + 多维度** = 一件事（如对同一份代码分析架构+质量+安全 → 一步 `llm_analyze`）
- **不同 action 或不同性质的产出** = 多件事，必须拆步。包括：
  - 「分析代码」+「生成报告」→ 两步（analyze + generate）
  - 「翻译全文」+「写摘要」→ 两步（generate + analyze）
  - 「重构代码」+「补写测试」→ 两步（两个 generate，产物不同）

**翻译与摘要/总结即使针对同一输入，也必须拆为两步**——翻译是 `llm_generate`（产出新文本），摘要是 `llm_analyze`（提炼要点），两者的 Skill Prompt 约束不同，合并会导致长度控制互相冲突。

#### 合并判定（机械规则，无业务例外）

写完全部步骤后，执行 lint 式检查：若同时满足——

1. `action` 相同；
2. 主输入字段引用在结构上相同（`llm_analyze` 比较 `inputs.content`、`llm_generate` 比较 `inputs.context`）：均为整段 `"{{{{变量}}}}"` 且变量名相同，或字符串字面完全相同；

则必须合并为一个步骤。

#### 后续步骤基于前序产物

若已产出合并后的分析结果（如 `full_analysis`），后续 `llm_generate` 的 `context` 应引用 `{{{{full_analysis}}}}`，**不要**再把原始大正文整段重传。原始超大正文对 LLM 通常只消费一次。

### 2.3 `instruction` 措辞

- 措辞应与用户需求的详细程度一致：用户只说「总结一下」→ 写「用 3–5 句话概括」；用户要求「详细」再写维度列表。
- 少用叠床架屋的形容词（「完整的、专业的、详尽的」连用易触发超长输出）。
- 泛泛的并列要求不超过 3 条。例外：同类分析合并时允许编号列出多个维度。
- 不要把「最终报告的章节清单」塞进单个 `llm_generate` 的 instruction——用 `file_writer` 内插组装。

#### 数字指标与任务规模

- 优先写「至少 1–2 个」等小而稳的下限，不鼓励产出膨胀。
- 对代码类生成，把可核对的硬要求写在 instruction 中（如 `test_` 前缀、`assert`、`pytest import`），verify 只检查这些客观条件。

### 2.4 多产物组装规则

当多个步骤的产物需要组合成最终文件时，**禁止默认用 `llm_generate` 做「汇总报告」**。直接在 `file_writer` 的 `content` 中用字符串内插拼装。

唯一例外：用户明确要求「用自然语言重新组织各部分内容」。

## 三、`llm_verify` 专项规则
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 3.1 定位：只做「客观可核对」的软门禁

`llm_verify` 的本质是「LLM 审 LLM」。它要在 `on_fail` 闭环里可靠收敛，前提是：`criteria` 中的每一条都应让 verify 模型仅靠模式/结构匹配就能判断，不需要「理解、评价、打分」。

**核心原则：权力与控制范围成反比。** `llm_verify` 若拥有「打回重做」的权力（`on_fail`），其控制范围必须清晰、确定、可客观核对；越主观的标准，越不应配 `on_fail`，否则极易进入死循环。

### 3.2 `criteria` 允许的条件类型

编写 criteria 前，先判断每条要求是客观可核对的还是主观评价的。只有客观条件允许写入 criteria：

- **结构存在性**：包含至少 N 个 section / 函数 / 端点
- **格式合规**：使用 pytest/unittest 框架；包含 import 语句；Markdown 格式正确
- **引用一致性**：import 路径符合 `src.` 前缀；特定标识符/字符串出现在产物中
- **占位符检查**：不含 `{{{{…}}}}`、`<待填写>`、`TBD` 等
- **基本完整性**：不为空；长度超过某阈值；不得只有标题/空壳

### 3.3 `criteria` 禁止的条件类型

以下属于主观评价，**不要**用作门禁条件（尤其不要配 `on_fail`）：

- 「质量达到生产/专业标准」
- 「译文准确/自然/流畅」
- 「测试充分覆盖关键场景」
- 「分析深入/全面」
- 任何带「足够、合理、恰当、充分、显著提升、专业级」等程度副词的条件

主观质量靠 `llm_generate` / `llm_analyze` 的 instruction 和 Skill Prompt 自身约束，不靠 verify 事后打分。

### 3.4 使用规则

- **每个 `llm_verify` 只验证一种产物**：`artifact` 只引用该产物的变量。不要把多种上游产物拼在一起「一次 verify 对照全部」。
- **「生成→验证」成对紧邻**：每个 `llm_verify` 紧跟在对应的 `llm_generate`/`llm_analyze` 之后，`on_fail` 只回跳该组的生成步。禁止交叉排列（会被校验器报 `OVERLAPPING_RETRY_LOOPS`）。
- **`llm_analyze` 默认不加 verify**：仅当（1）用户明确要求，或（2）同一份分析被多个后续步骤长期依据时，才在分析步后加 verify。
- **需求为「基于给定输入做修改/分析/重构」时**：建议加 verify，criteria 写结构/占位符/明显跑题等客观条件。
  - 代码类：criteria 写可执行性、import、assert、占位符等单文件内可核对项。
  - 纯文本类：criteria 写「不含占位符、不为空」等。

## 四、YAML 输出语法规则
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下与业务场景无关，只约束工作流 YAML 怎么写。

### 4.1 `step.action` / 技能名

只能使用「可用技能清单」中出现的技能名称，逐字一致。即使你认为存在更「自然」的名字，只要清单中没有，就禁止使用。

### 4.2 `step.inputs` 的值形态

三种合法写法：

- **纯变量引用**：整段恰好为 `"{{{{variable_name}}}}"`。保留类型。
- **字符串内插**：含一处或多处 `{{{{variable_name}}}}`。Runner 逐个替换为 `str`。
- **字面量**：不含 `{{{{...}}}}` 的固定字符串。

**禁止用单花括号**引用上下文变量。必须用双花括号 `{{{{var}}}}`。

**读取素材**：统一使用 `file_reader`。传入 `path`/`file_path`（单路径）或 `paths`（列表/逗号分隔）。引擎自动识别单文件、目录、`.zip`。

### 4.3 变量名前后一致

后续步骤 `inputs` 中引用的变量名必须与前序步骤 `outputs` 的 key 逐字符一致。步骤超过 4 个时，每写一步前先核对当前变量集合。

### 4.4 `step.outputs` 必须是字典

- **key**：写入上下文的变量名（自定义，snake_case）。
- **value**：技能 Pydantic 输出模型上的字段名，必须与技能清单逐字一致。
- 同一技能多步串联时，每步用不同的 key。

### 4.5 `on_fail` 规则

- 目标必须是需要重新执行以修复失败的那一步的 ID。
- 必须严格小于当前步骤 ID。
- **多组「生成→验证」成对紧邻**，禁止交叉排列。
- **`llm_verify` 一次只验一种产物**，`on_fail` 只回跳对应的生成步。
- **`criteria` 只写客观条件**。

### 4.6 其他

- 变量名一律 snake_case。
- 文件路径放在 `file_reader`/`file_writer` 的 inputs 或工作流 inputs 中。
- 子工作流：仅当用户明确要求时使用 `sub_workflow`。
